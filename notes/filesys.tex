\section{File systems}
\begin{itemize}
    \item Sector 0 Master Boot Record, partition table, partitions
    \item Partition: boot block, superblock, free space mgmt, inodes, ...

    \item BIOS executes MBR program. MBR program finds active partition (where OS is), runs boot block,
        boot block loads OS into RAM.

    \item Contiguous allocation: removing files creates holes, can't grow files

        Good for CD-Rom b/c never change memory

    \item Linked-list allocation: allocate in blocks

        files are scattered through the system: pointer to physical block and next physical block

        sequential (not random) access b/c have to go through block pointers

        seeing which block is hard b/c block size is not a power of 2 b/c of pointer

    \item FAT: file allocation table (windows)

        start of partition

        in directory structure: store 1st block num. then fat stores next block, table[block idx] = next idx.

        FAT[last block] = -1

        requires huge FAT: 2GB disk, 1kb block = 2 M entries

        fixed size based on partition size and block size

        Disk access to follow idxs and collect all blocks nums, then disk access to read.

\end{itemize}

I-node
\begin{itemize}
    \item file attributes

    \item first 10 addresses of data blocks (based on avg file size)

    \item address of disk block containing additional pointers to indirect blocks

    \item Indirect blocks: point to data blocks or other pointer blocks

        1 single indirect block + 1 double indirect block + 1 triple indirect block \\

        ex: 10 initial addr, 1kb blocks, 4-byte pointer to data block

            max file size = $10*1Kb + (1024/4)*1Kb + (256^2)*1Kb + (256^3)*1Kb$
\end{itemize}

file allocation size matters for spinning disk
\begin{itemize}
    \item more blocks = more seeking and track changing
    \item allocation unit = 4 sectors (blocks), less blocks you have to seek
    \item increasing block size increases data rate (read speed) but decreases storage usage.
\end{itemize}

Disk w/ 1mb per track. rotation time = 8.33ms. avg seek time = 5ms. time to read block of k bytes?
\begin{itemize}
    \item seek time = 5ms
    \item avg rotation time before seeing data = 8.33/2 ms
    \item total time = 5 + 8.33/2 + 8.33 * (k / 1024) / 1024\\
\end{itemize}

\section{Backups}
\begin{itemize}
    \item Physical dump: copy everything (all blocks)

        no incremental dump

        cannot restore individual files

        dumps unused blocks

        dumps bad blocks: disk scanning tool to find areas of hard disk that fail read/write test

    \item Logical dump: only dump things that changed

        bitmap of all dirs: unmark dirs w/out modified files

        dump only dirs and modified files, then you can reconstruct
\end{itemize}

file system consistency
\begin{itemize}
    \item deleted file: mark block free, update inode, free block list
    \item what if crash before adding to free block list?
    \item go through inode structure, build list of blocks in use

        blocks in use and free blocks should be complementary

    \item block allocated to 2 files: make 1 file use a free block
\end{itemize}


\section{Caching}
\begin{enumerate}
    \item Caching disk blocks: buffer cache hash table: head = LRU, tail = MRU

        system can crash before writeback, then changes cached in RAM, never flushed to HDD

        LRU: inodes rarely referenced 2x, so will be in cache for a long time, danger for crash

        modified LRU: if block not needed again / critical file, then go to start of list (LRU)

        windows write-through cache: write to disk immediately, but also cache for read

        linux sync / windows flush file buffers: flush to disk

    \item block read ahead: bit for sequential / random access

        if sequential: keep reading block ahead (to save on having to seek + find start again)

    \item reduce disk arm motion: cylinder group w/ own inodes ot keep inodes + data close

        SSD: constant read/write time, spread out data for lifetime

    \item defragmentation: consolidate HDD blocks

    \item log structured file system: most disk ops are writes

        circular log: repeat (inodes, datablocks)

        remove a file: remove from dir, release inode to free pool, release disk block to free pool

        cleaner thread: defragment
\end{enumerate}
