\section{Devices}
Device controller: between driver and hardware, error correction
\begin{enumerate}
    \item serial bits to byte block
    \item error correction
    \item copy data to main memory
    \item communicating with controllers

        control register assigned IO port: mobo has specific ports, device can only connect to certain ports

        memory map: maps control register to memory space
\end{enumerate}

interrupts:
\begin{enumerate}
    \item dedicated line (pin) to interrupt controller on mobo
    \item mouse click, signal saves in control register
    \item controller tells CPU interrupt raised (which device)
    \item CPU calls instruction to read interrupt port (IN AX, 15)
    \item to send instuction: write in serial to controller port (OUT 100, AX)
    \item device controller reads bytes, then prints
\end{enumerate}

memory map:
\begin{enumerate}
    \item write in C, don't need to use asm insructions
    \item addresses in RAM are mapped to IO port addr
    \item device driver can be writeen in C (no need for special ASM instruction, just files)
    \item os ensures control registers aren't in user virtual addr

    \item problem with caching: CPU uses cached value, doesn't go to RAM (but voltaile b/c IO)

        set cache disabled bit in memory mapped IO page

        PCI brige b/t CPU and main memory, forwards memory requests to peripheral device bus,
        that area in main memory has to be empty
\end{enumerate}

direct memory access
\begin{enumerate}
    \item fread(``x.txt", \&buf, 100)
    \item file system uses inode structure to find ``x.txt", finds data blocks
    \item OS writes magic num + block number + buf to disk controller to read
    \item disk controller reads to internal buffer, error check, raises interrupt
    \item OS copies internal register to main memory
    \item problem: naive reading will cause interrupt per block read

        CPU tells dma controller to interrupt only after all blocks are read,
        disk controller copies block to main memory notifies dma buffer,
        dma controlelr raises interrupt at the end. must lock bus to copy to main memory

        reduces \# of interrupts to CPU

    \item word at a time mode: cycle stealing, locks bus
    \item burst mode: locks bus, copies many blocks at once
    \item device to device copy using dma internal buffer instead of main memory
\end{enumerate}

t1 ns to acquire bus, t2 ns to transfer 1 word over the bus.
1000 words, ack transfer also required sending a word over the bus
\begin{itemize}
    \item 1 word at a time: 1000 * 3 * (t1 + t2), acquire + command, acquire + send, acquire + ack

    \item burst mode: (t1 + t2) + t1 + 1000*t2 + (t1+t2), acquire + command, acquire, 1000 copy, acquire ack
\end{itemize}

interrupt
\begin{enumerate}
    \item printer sends signal over bus to interrupt controller
    \item interrupt controller tells CPU interrupt, gives interrupt address, disable interrupts
    \item CPU dumps current pc, registers onto stack
    \item jumps to interrupt vector table[interrupt address], runs interrupt service
    \item save pc and registers to process table
    \item do actual service, read specific register on printer via IO
    \item go to OS, OS schedules, reload registers and PC
\end{enumerate}

pipelining and interrupts
\begin{enumerate}
    \item interupt when some instructions are partially done (imprecise interrupt)j
    \item lets some instructions fully finish, terminate other ones (precise interrupt)
\end{enumerate}

software interrupts
\begin{enumerate}
    \item INT instruction (from user level)
    \item gives control to OS (printf)
    \item instruction: INT X, give syscall args.
    \item load file, sys read, read file bytes 
\end{enumerate}

io software
\begin{enumerate}
    \item sync (blocking) vs async: OS makes interrupt driven look blocking (easier programming)
    \item io buffering
\end{enumerate}

programmed io vs interrupt-driven
\begin{enumerate}
    \item programmed io: busy waiting
    \item interrupts

        user calls print system call, enable interrupts, put a char into printer

        on printer interrupt, send another character

        printer interrupts on every character, use DMA controller

    \item DMA: user sets up dma and sleeps, dma handles printing, CPU interrupt unblocks user at the very end
\end{enumerate}

magnetic disks
\begin{enumerate}
    \item seach sector has preamble, data, ecc
    \item cylinder skew: shift 0s in sectors so that when you move the needle the next 0 sector is right there
    \item dma buffer size: \# sectors you can store in dma so you don't have to wait another rotation

        then ``interleave" during formatting so that once you copy dma to ram, next section is under needle

        modern disks just read whole track into buffer

    \item disk arm scheduling: first come first serve

        shortest seek first: starvation can happen if close disk block requests keep coming in

        elevator algorithm: just make sweeps in a single direction and reverse. worst case is 2 sweeps,
        do better by sweeping right, then starting from left again

    \item bad sector

        spare sector swapping, internal tables for mapping, breaks contiguous read/write

        shift all sector numbers so still contiguous

    \item seek error: need to recalibrate mechanical movements
\end{enumerate}

stable storage
\begin{enumerate}
    \item disk block: preamble (disk block number) + data + ecc
    \item ecc is some polynommial code???

    \item 2 identical disks:

        stable write: write 1, reread 1, write 2, reread 2

        stable read: read 1, check ecc. if wrong, try disk 2

        crash recovery: overwrite other one with correct ecc. both correct but different, then take newer
\end{enumerate}

Redundant Array of (Inexpensive / Independent) Disks (RAID)
\begin{itemize}
    \item level 0: stripping = distribute data over multiple drives.

        write file parts to multiple disks instead of 1 disk.
        
        write faster, but more frequently crashes

    \item level 1: 2 duplicate raid level 0s

        read: 2x more efficient b/c have 2x the disks to read from

        write: same time b/c write to each at the same time

        more reliable b/c backup

    \item level 2:

        39 disks: 32 bits + 6 parity + 1 word partiy

        way more disks: faster read

        rotation sync overhead

    \item level 3: 1 parity disk.

        problem: can only detect 1 bit error

        1 bit error correction for disk crash

        1 bit error detection for random error

    \item level 4: strip level parity

    \item level 5: distribute parity across drives

    \item bit and strip distribution: split disk up into bits and strips, assign to disks

    \item binomial failture: n disks, recover from 1 disk failure

        $X \sim \text{Binom}(n, p)$

        $P(\text{fail}) = P(X \ge 2) = 1 - P(X = 0) - P(X = 1) = 1 - (1-p)^n - \binom{n}{1} p^1 (1-p)^{n-1}$
\end{itemize}

Clock
\begin{itemize}
    \item crysal oscillator generates pulse
    \item one shot mode: holding register loads counter, counter is decremented each pulse, interrupt when counter = 0
    \item square wave mode: repeatedly copy register to counter, decrement, interrupt

        periodic interrupts = clock ticks

    \item interrupt launches clock driver:

        maintain day time

        preventing procs from running too long: scheduler initializes counter w/ process CPU quantum in ticks

        accounting CPU usage: only track CPU time used by proc, start new timer on proc start.
            increment counter for the current process

    \item alarm sys call

        ex: ack must be received in certain time. if lost, then must resend packet

        duplicate packets b/c server thought lost: TCP has sequence number

        option 1: table with alarm times, driver checks periodically, raises alarm if curr time > alarm.
            can make you late

        option 2: linked list of ticks to next alarm, periodically check, raise alarm

    \item watchdog: self-monitor parts of system so no hang (common in embedded)

        timer periodically sets 1, system periodically clears, if not cleared then reset system
\end{itemize}

soft timers
\begin{itemize}
    \item polling: busy waiting

        interrupt: no waiting but overhead from context switch

    \item whenever interrupted by OS (syscall, TLB, page fault): check timer + do routine

    \item saves on interrupts, just piggybacks off of OS interrupt

    \item secondary backup low-freq timer for low-syscall periods
\end{itemize}

power management
\begin{itemize}
    \item most power intensive: display

        OS automatically rearranges windows to activate minimum zones

        not used

    \item Dynamic Voltage Scaling (DVS)

        run processor at lower voltage and lower speeds during off-peak times

        speed / 2 = voltage / 2 = power / 4
\end{itemize}
