static relocation:
* loader function goes through program, adds program start addr to all addresses
* slow, must get recomputed every time

base and limit registers

* gives each proc a separate addr space
* hardware adds addr to base register, checks that under limit register

# how to fit procs in memory?
swapping: moving procs back and forth from HDD
* memory holes: memory compaction, copying procs to make contiguous
* leave space for procs to grow (to allocate more dynamic memory)

not enough space: move to bigger hole, suspend until space opens, kill proc


garbage collector (Python): frees up memory not in use (local scope)
* C: no GC, programmer manually manages memory

RAM is split into 4kb chunks
* each chunk is represented by bit in bitmap
* malloc: scan for chunk of large enough size
* process table: proc, start, size

linked list: process start length | hole start length
* split holes into used and unused in 
* free: collapse consecutive holes

memory allocation algoirthms
* first fit: scan for hole large enough to fit proc, best
* next fit: put proc in last index in list
* best fit: find hole just big enough, can end up with many tiny holes
* worst fit: find largest hole
* quick fit: multiple lists, <4kb linked list, <8kb linked list, have to move holes around when allocating


overlays

virtual memory
* program on disk, load pages to RAM
* clean/dirty vridvpages, must write-back pages to RAM
* page size: 4kb

page table
* virtual page # -> page size
* virtual address (within page) -> physical address (could be in other page)
* pagefault: OS has to find page in disk, moves page to ram (throws out a page)
    * launches page fault handling routine, OS runs, suspends P1, runs P2, OS schedules P1 again, P1 tries to get page again

32k ram, byte addressable (32k byte entries)
* need 15 bits
* each page frame is 4k bytes (12 bits)
* so 3 bits to address the page frame (32k / 4k = 8)
* 12 bits to address byte in page (offset in page)

12 byte ram, 4 byte pages
* 4 bit address: 2 page number bits, 2 offset bits
0111 -> page 01 = 1, offset 11 = 3 -> 3rd byte inside page 1

32 byte program
page table needs 8 entries (4 byte pages)
* virtual page mapped to physical page

virtual page number -> physical page | virtual page offset = physical page offset

virtual addr 000|11, page table: virtual 0 -> physical 1
physical addr 01|11

access memory:
* try to get memory at virtual address
* calculate page number, look in page table
* if page not in page table: pagefault, writeback if existing page is dirty, get page
* split virtual address = virtual page | offset, translate into physical page | offset using page table virtual page -> physical page

paging b/c can have arbitrarily large program (virtual addr space) map onto smaller physical memory
* small RAM, load program form disk


MMU (Memory Management Unit) b/t CPU and memory, converts virtual addr -> physical addr

page table: virtual page idx -> physical page frame
* virtual addr = page idx | offset
* MMU checks page_table[page_idx] present/absent bit: if absent, then page fault, go fetch page frame from HDD
* physical page idx = page_table[page_idx]
* physical addr = physical page idx | offset


32-bit virtual addr. 4kb page size. 512mb ram.
* 4kb = 2^12, 12-bit offset
* 32 - 12 = 20-bit page table idx. 2^20 virtual pages.

virtual addr: 13025 = page table idx 0000 0000 0000 0000 0011 | offset 0010 1110 0001

page table entry
* present/absent bit, physical page frame idx
* referenced bit: 0 init, page accessed, R-bit is cleared periodically for freshness
* modified bit: 0 init, page written to, must write back to HDD when replacing page
* protection bit: RWX permissions


TLB: translation lookaside buffer
* cache of page table in MMU
* small number of virtual page -> page frame mappings
* valid bit (if empty is not empty)
* TLB miss: virtual page not in TLB, look in RAM page table
    * soft miss: RAM page table says page is in RAM
    * hard miss: page not in RAM, go to HDD
* context switch: flush TLB, b/c each proc has its own virtual address space (different page table)


each process has an entire copy of virtual address space? huge?
* most pages in virtual addr space are unused (small code + data compared to virtual addr space)
* only store vitual pages in use in page table

Multi-level page table
* Virtual addr = | level 1 page table idx | level 2 page table idx | offset |
* level 1 page points to level 2 pages, only used meta-pages are created in top level page table
* saves a lot of space b/c only store meta-pages in use, but requires 2 memory accesses, level 1 -> level 2 -> phyiscal RAM addr


32-bit virtual address, 4kb page size
program + data space < 4kb, stack < 4kb

# page table entries for 1 level mapping: 2^32 / 2^12 = 2^20
2 level mapping:
* valid entries: 2 meta (1 program + data, 1 stack, both under 4mb) + 2 actual (both < 4kb)
* page table entries: 1024 meta + 2 * 1024 (program + data, stack) = 3*1024


4kb page size, 64-entry TLB, int array
for what values of M (array step), N (array size), will you get a TLB miss for every iteration
* TLB miss if: step larger than page size, int array so 4kb / 4 bytes = 1024, M = 1024
* 64-entry TLB, so need at least 64 misses, N = 1024 * 64, aka 64 * 4kb page size / 4 byte int


TLB: 1024 entries, access time is 1 clock cycle (1ns)
Page table entry: access time is 100 clock cycle
Avg page replacement (page HDD -> RAM): 6 ms
TLB hit rate is 99%
0.01% page fault (requires page replacement)

effective addr translation time = (0.99 * 1ns TLB hit) + 0.01 TLB miss * (0.99 * 100ns RAM + 0.01 * 6ns HDD)


inverted page tables
* normal page table: many virtual pages -> few physical pages  (very sparse, but constant lookup)
* idea : few physical pages -> virtual pages     (iterate through physical pages, check if virtual page, dense, memory efficient, slower lookup)

hash table: hash on virtual page -> linked list of (virtual page, process id, physical page)
* # entries in hash table = # physical pages
